# Format Prompt — 분석 .md 형식 명세

> 이 파일은 Writer 서브에이전트가 `analysis.md`를 작성할 때 따르는 **형식·흐름 명세**다. 사용해보면서 직접 편집하거나, 메인 Claude에게 "다음부터는 ~이렇게 해줘" 라고 피드백하면 자동 갱신된다.

---

## 1. 전체 톤

- 1인칭 학습 노트 스타일. "내가 이 논문을 다시 봤을 때 빠르게 이해할 수 있도록" 정리.
- **전문 용어는 영어 원어 그대로 쓴다.** 어색한 직역(예: "행렬곱", "잠재 표현", "주의 기법", "잔차 연결")으로 옮기지 않고 `matrix multiplication`, `latent representation`, `attention`, `residual connection` 처럼 원어 유지. 단, 처음 등장 시 1줄 짜리 한국어 직관 설명을 괄호나 짧은 부연으로 붙인다.
  - 예: "linear self-attention (시퀀스 길이에 비례하지 않고 상수 비용으로 작동하는 어텐션 변형)"
  - 한국어로 굳어진 일반 표현(예: "학습", "추론", "매개변수")은 한국어 그대로 OK.
- 영문 인용·고유명사·메서드 이름·라이브러리 이름은 원문 그대로.
- "본 논문은 ~을 제안한다" 같은 학술적·수동적 표현 지양. "저자들은 ~한다" 또는 "이 논문은 ~한다"처럼 능동·평이하게.

## 2. 문서 구조 (이 순서·이 섹션을 꼭 지킬 것)

```markdown
# {논문 제목}

> {arXiv URL/DOI}  
> {저자 1, 저자 2, 저자 3 et al.} · {발행 연도/장소}

## TL;DR
- 한 줄로 무엇을 한 논문인지
- 한 줄로 핵심 인사이트
- 한 줄로 결과의 임팩트
(불릿 3개 고정)

## 사전 지식
이 논문을 이해하기 위해 알아야 하는 *기존 개념·선행 기법*을 짧게 정리.
2~4 문단 또는 항목별 짧은 단락.
- 이 논문이 빌리거나 확장하는 핵심 빌딩 블록 (예: MesaNet 정리라면 `linear self-attention`, `Mesa layer`, `gated linear attention` 같은 것)
- 각 항목은 "이게 뭔지 1~2줄 + 이 논문에서 어떻게 쓰이는지 1줄"
- 이 섹션은 *논문이 직접 정의하는 새 개념*은 다루지 않는다. 그건 "핵심 아이디어"의 몫.
- 선행 지식이 거의 없는 단순한 논문이면 1문단으로 짧게 끝내거나, 정말 필요 없으면 "(생략)"으로 표시.

## 풀어야 했던 문제
2~4 문단. 이 분야의 기존 한계가 무엇이었고 왜 그 한계를 부쉈어야 했는지. 
저자들의 동기를 추적할 수 있게.

## 핵심 아이디어
1~2 문단으로 "이 논문이 해결책으로 제시하는 한 줄 요지"를 먼저 명시.
그 다음에 그 아이디어가 왜 작동하는지 직관적 설명.
필요하면 아키텍처/플로우 그림 1장 임베드. 
{핵심 그림이 있으면 여기에 ![](figures/all/figure_X.png) 형태로 삽입}

## 방법 / 접근
2~5 문단. 알고리즘/수식/모듈을 단계별로. 
- 수식은 LaTeX 블록으로
- 핵심 변수는 정의를 인라인으로 명시
- 그림은 본문에 자연스럽게 인용 (본문에서 "Figure 2처럼 ..." 같이)

## 실험과 결과
- 어떤 데이터셋/벤치마크 위에서
- 어떤 베이스라인과 비교
- 어느 메트릭에서 얼마나 (표로 정리)

| Metric | This Work | Baseline | Δ |
|---|---|---|---|
| ... | ... | ... | ... |

핵심 결과 그림이 있으면 표 아래에 임베드.

## 왜 이게 중요한가
1~2 문단. 이 논문이 후속 연구에 미친 (또는 미칠) 영향. 
실용적 관점에서 무엇이 가능해졌는지.

## 한계와 의문
- 저자들이 명시한 한계
- 내가 읽으면서 느낀 의문 (있으면, "내 의문:" 으로 표시)
```

## 3. 피규어 사용 규칙

- **임베딩 개수**: 핵심 피규어 2~5장이 일반 가이드. 단 framework/architecture 그림은 예외 (아래).
- **Framework/architecture 그림은 빠짐없이 전부 임베드한다.** 같은 모델의 abstract schematic과 concrete diagram이 둘 다 있으면 둘 다 넣는다 (예: Fig 1 = encoder/prior/decoder 박스 그림 + Fig 2 = KL formulation schematic이라면 둘 다). 판단 기준:
  - 캡션에 "overview", "architecture", "schematic", "framework", "pipeline", "model diagram" 같은 단어가 들어가거나
  - 모델 컴포넌트(Encoder, Decoder, Prior 등)의 박스/모듈을 그림으로 표시하거나
  - 입력 → 출력 흐름을 시각화한 그림
  → 이 중 하나라도 해당하면 framework 그림으로 보고 모두 포함.
- **나머지 그림** (실험 결과 plot, ablation, 정성적 sample 등)은 본문에서 가장 자주 언급되거나 핵심 결과/비교를 보여주는 것으로 2~3장 선별.
- **위치**: 항상 그림이 설명되는 본문 *바로 다음* 줄에 삽입. 멀리 떨어뜨리지 않기. Framework 그림 여러 장은 "핵심 아이디어" 또는 "방법 / 접근" 섹션에 연속 배치 (각 그림 사이에 1~2 문장의 짧은 본문이 들어가도 OK).
- **캡션**: 마크다운 이미지 문법 그대로 (`![Figure 2: 캡션](figures/all/figure_2.png)`). 너무 긴 원본 캡션은 핵심만 한국어로 요약.

## 4. 수식·코드

- **수식**: KaTeX 호환 LaTeX. 인라인 `$...$`, 블록 `$$...$$`.
- 정의되지 않은 기호 금지. 변수 처음 쓸 때 정의.
- **의사코드**: 필요할 때만, ` ```python ` 펜스 안에 (구문 강조용).

## 5. 길이

- 전체 ~1000–1800 단어 (한국어 기준 ~2000–3500자) 권장. "사전 지식" 섹션 추가로 기존 가이드보다 살짝 늘어남.
- TL;DR과 표는 단어 수에서 제외.
- 사전 지식이 거의 없는 단순한 논문이면 그 섹션을 1문단으로 줄이고 권장 길이도 비례해서 줄어든다.

## 6. 렌더 외관

- **줄 길이**: 본문 한 줄이 페이지를 가로지를 정도로 길어지면 안 된다. 한국어 기준 한 줄에 *대략 40~55자* 정도가 들어가도록 본문 영역이 좁혀져야 한다. 영문 기준으로는 한 줄 ~75자.
- 이 제약은 `scripts/render_md.py`의 CSS `body { max-width: ... }`로 강제된다 (작성자가 직접 제어할 수 있는 변수는 아니지만, 형식 명세의 약속으로 못 박아둔다). max-width는 A4 콘텐츠 폭(2cm 여백 기준 ~642 CSS px)보다 *작게* 잡혀야 — 즉 max-width ≤ 560px 정도 — 좌우에 여백이 생기며 wide한 인상이 사라진다.
- 본문 폰트는 11pt 기준에서 줄당 글자 수가 너무 많지 않게. 표·코드 펜스·수식 블록도 동일한 max-width 안에 들어와야 한다.
- 페이지 마지막에 큰 빈 여백이 남는 것 자체는 issue 아님 (자연스러운 끝맺음).

## 7. 금지 사항

- "본 논문에서는", "사료된다", "할 수 있을 것이다" 같은 학술 번역투
- 의미 없는 부연 ("주목할 만한 점은", "흥미롭게도")
- **전문 용어를 어색한 한국어로 직역**. 예: "행렬곱"(❌) → `matrix multiplication`(✓), "은닉 상태"(❌) → `hidden state`(✓), "기울기 강하"(❌) → `gradient descent`(✓). 한국어 직관 설명은 처음 등장 시에만 1줄 부연으로.
- 원문에 없는 수치/주장 삽입 (환각 금지)
- 본문에 한 번도 인용되지 않은 그림 삽입
